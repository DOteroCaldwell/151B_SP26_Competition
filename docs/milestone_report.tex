\section{Introduction}
\label{sec:intro}
Introduce the problem that you are trying to solve and write 1-2 paragraphs for each of the sub-sections.  Provide a high-level summary in this section instead of detailed descriptions.
\paragraph{Problem Definition.}
The task is to maximize mathematical reasoning accuracy on a unified benchmark of 1,126 questions: 375 multiple-choice (select a letter A--J) and 751 free-form (produce numerical/symbolic answers or multi-part lists). Model output must place answers inside \texttt{\textbackslash boxed\{\}} delimiters; evaluation uses exact matching for MCQ letters, symbolic equivalence for expressions, and numeric tolerance $10^{-8}$ for floating-point values. For multi-part questions (414 of the free-form set), \emph{all sub-answers must be correct} for credit. We use only \texttt{Qwen/Qwen3-4B-Thinking-2507} (4B parameters) with no external APIs, calculators, or code interpreters at inference time. The metric is unified accuracy: (correct / total) questions, with equal weight regardless of difficulty or type.

\paragraph{Problem Significance.}
Small, open-weight models with strong mathematical reasoning enable deployment in resource-constrained environments (mobile devices, edge servers, offline systems). Scientific research, engineering design, finance, and education rely on reliable quantitative reasoning from LLMs. Improving 4B-parameter models is valuable because: (1) inference latency and cost drop significantly vs. 70B+ models, making deployment practical, (2) interpretability and transparency increase with smaller scale, and (3) efficient reasoning strategies discovered here often generalize across model sizes. Success would demonstrate that careful prompt design and fine-tuning can extract strong reasoning capability without relying on massive model scale or external solvers.

\paragraph{Technical Challenge.}
Multiple challenges span data, model, implementation, engineering, and evaluation: \textbf{(1) Format compliance:} The model must reliably output answers in \texttt{\textbackslash boxed\{\}} delimiters. Baseline testing on 20 questions reveals 47\% of errors are formatting failures (missing boxed wrapper or invalid letters), not mathematical reasoning errors—indicating that instruction-following and output formatting are severe bottlenecks. \textbf{(2) Multi-part reasoning:} The model must parse implicit `[ANS]` placeholders, output sub-answers in the correct order, and understand that a single incorrect sub-answer causes the entire question to fail. This is structurally harder than single-answer tasks. \textbf{(3) Domain diversity:} The benchmark spans algebra, calculus, complex analysis, probability, and statistics across difficulty levels; no single prompt is optimal for all, requiring domain-specific tuning or adaptive routing. \textbf{(4) Computational efficiency:} INT4/INT8 quantization is mandatory. Inference throughput directly limits the feasibility of sampling strategies (majority voting, beam search) needed to improve accuracy. \textbf{(5) Evaluation complexity:} The \texttt{judger.py} framework must handle symbolic simplification (SymPy), numeric tolerance with floating-point precision, and structural matching. Bugs in evaluation mask real improvements or hide real regressions. 


\paragraph{Contributions.}
The starter code provides data loading, inference scaffolding, and the \texttt{judger.py} evaluation framework. Its limitations: (1) generic system prompts that do not explicitly enforce formatting constraints, (2) no error analysis to identify failure modes or problem-category trends, (3) single-pass inference only (no sampling or majority voting), and (4) no prompt iteration framework. To date, our contributions are:

\begin{itemize}
  \item \textbf{Comprehensive error categorization:} Mini-baseline testing on 20 questions revealed three error types: MCQ with no valid letter (7 errors, 47\%), free-form with no \texttt{\textbackslash boxed\{\}} wrapper (7 errors, 47\%), and MCQ with wrong letter (1 error, 7\%). This isolation of formatting failures from reasoning failures directly motivates Phase 2 prompt improvements.

  \item \textbf{Dataset precision audit:} Confirmed ground-truth answers span 0--16 decimal places with no uniform rounding. This validates the use of numeric tolerance ($10^{-8}$) in \texttt{judger.py} rather than exact floating-point matching.

  \item \textbf{Reproducibility infrastructure:} Documented full baseline configuration (vLLM INT8 quantization, batch size 4, $\text{temperature}=0.6$, $\text{top}_p=0.95$, $\text{top}_k=20$, $\text{max\_tokens}=2048$), hardware specs (NVIDIA RTX PRO 6000 Blackwell), estimated wall-clock time (13 hours for 1,126 questions), and a reproducible error-logging framework.

  \item \textbf{Performance-by-category insights:} Mini-baseline shows single-part free-form answers achieve 75.0\% accuracy (3/4) when formatted correctly, while multi-part answers achieve only 14.3\% (1/7), indicating that multi-part reasoning and formatting are the bottlenecks, not single-answer reasoning capability.
\end{itemize}

\paragraph{Error Analysis.}
Of 15 incorrect responses, 14 (93\%) are formatting failures: MCQ\_NO\_VALID\_LETTER (7 errors), FREE\_FORM\_NO\_BOX (7 errors). Only 1 error (7\%) is MCQ\_WRONG\_LETTER. This indicates that output formatting and instruction-following are severe bottlenecks, not mathematical reasoning. The 75\% accuracy on single-part free-form vs. 14.3\% on multi-part suggests that multi-part answer structure (ordering, comma-separation) is poorly understood by the model with default prompts.
